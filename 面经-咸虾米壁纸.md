# 咸虾米壁纸 — 面经

> 技术栈：UniApp (Vue3) + 微信小程序/H5/App 多端适配 + SCSS

---

## 1. 项目简介（简历可用）

基于 UniApp + Vue3 开发的跨端壁纸应用，一套代码编译至微信小程序、H5 及 App 三端，支持 Banner 轮播、分类精选、专题推荐、搜索、排行榜、壁纸全屏预览（swiper + 前后三张预加载）、评分、下载保存到相册与用户中心等核心模块；核心难点在于多端导航栏高度动态适配、条件编译处理平台差异、本地缓存传递大图列表以及分享链路打通。

---

## 2. 简历 bullet（4～6 条）

- 负责基于 UniApp (Vue3) 构建跨端壁纸应用，一套代码覆盖微信小程序、H5、App 三端，通过条件编译（`#ifdef` / `#ifndef`）处理导航栏、客服入口、下载保存等平台差异逻辑。
- 实现自定义导航栏组件，结合 `uni.getSystemInfoSync` 与 `uni.getMenuButtonBoundingClientRect` 动态计算状态栏、胶囊按钮及标题栏高度，解决 iOS/Android 多机型适配问题（待测）。
- 设计壁纸预览页「前后三张」图片预加载策略，利用 `swiper` 滑动事件动态维护 `readImgs` 渲染白名单，降低大图全量加载带来的内存与网络开销（待测）。
- 封装通用请求模块 `utils/request.js`，统一拼接 BASE_URL、注入鉴权 header，并按 `errCode` 分流处理：0 正常返回、400 弹 Modal、其余 Toast 提示，实现全链路错误兜底。
- 使用 `uni.setStorageSync` 建立「列表页 → 预览页」本地缓存通道，绕过 URL 传参长度限制；在页面 `onUnload` 生命周期中主动清理缓存，避免 Storage 残留（待测）。
- 打通小程序分享链路，在首页、分类列表、预览页等核心页面配置 `onShareAppMessage` 与 `onShareTimeline`，支持带参分享与分享后单页恢复；并在近期上新模块中实现广告位动态插入与条件编译展示（待测）。

### 主题一：跨端适配与自定义导航栏（高频）

#### 主问 1：你在咸虾米壁纸项目中做了自定义导航栏，小程序不同机型、不同平台（微信/抖音/H5）高度差异很大，你是怎么保证导航栏不会和系统胶囊按钮重叠的？

> 口播：我在这个项目里负责自定义导航栏的高度适配，这是跨端开发里最直观也最容易踩坑的地方。情境是微信小程序的胶囊按钮在不同机型上位置不一样，iPhone 和 Android 的状态栏高度也不同，而抖音小程序又有自己的原生导航栏逻辑，H5 则完全没有胶囊按钮。我的任务是让导航栏在所有端都能正确占位，不被系统 UI 遮挡，也不出现多余的空白。行动上，我首先在 `utils/system.js` 里封装了一套布局计算函数：先用 `uni.getSystemInfoSync` 拿到 `statusBarHeight` 作为状态栏高度；再用 `uni.getMenuButtonBoundingClientRect` 获取胶囊按钮的 `top`、`bottom`、`right`、`width`，从而算出 `titleBarPadding` 和 `titleBarHeight`。对于非微信小程序端，比如 H5 或 App，因为拿不到胶囊按钮信息，我做了兜底：H5 的 `statusBarHeight` 直接给 0，`titleBarPadding` 默认给 8px，`titleBarHeight` 取计算值与平台最小值（iOS 48px / Android 44px）的较大者。在 `components/custom-nav-bar/custom-nav-bar.vue` 中，我把状态栏和标题栏用 `:style` 动态绑定高度，并在组件下方放一个等高的 `fill` 占位块，防止页面内容被导航栏盖住。针对抖音小程序，我通过 `#ifndef MP-TOUTIAO` 把自定义导航栏整个隐藏掉，避免和它的原生导航栏冲突。结果是首页、分类页、用户中心在多端测试中没有出现重叠或错位，导航栏高度随系统动态变化。

##### 追问 1.1：如果 `uni.getMenuButtonBoundingClientRect` 在某些特殊小程序平台（比如钉钉小程序）上获取不到，你的代码是怎么兜底的？

> 口播：这是一个真实遇到的边界情况。在 `utils/system.js` 的 `getMpMenuRect` 函数里，我先判断平台是否是 `mp-weixin` 或 `mp-alipay`，然后尝试调用 `uni.getMenuButtonBoundingClientRect`。如果这个方法不存在或者返回异常，我会走 else 分支做兜底：把 `mpMenuHeight` 设为 `SYSTEM_INFO.titleBarHeight - getTitleBarPadding() * 2`，`mpMenuMargin` 给 10，`mpMenuWidth` 给 80。这些默认值是基于支付宝小程序的 `titleBarHeight` 反推的经验值，能保证导航栏在大多数场景下不会偏差太多。同时，我在 `getTitleBarHeight` 里还加了一层最小高度保护：如果计算出来的高度小于 iOS 48px 或 Android 44px，就取最小值。这样即使胶囊按钮信息完全拿不到，导航栏也不会塌陷成一个很窄的条，导致标题被截断或者点击区域过小。

##### 追问 1.2：H5 端没有胶囊按钮，你的自定义导航栏在 H5 上表现和微信小程序一致吗？如果不一致，你是怎么处理的？

> 口播：不一致，而且我刻意让它们不一致。情境是 H5 运行在浏览器里，顶部是浏览器自己的地址栏和标签页，没有小程序胶囊按钮的概念。如果我在 H5 上也保留一个很高的自定义导航栏，会造成顶部空间浪费，用户体验反而变差。我的处理方式是：在 `utils/system.js` 的 `getStatusBarHeight` 里，判断 `SYSTEM_INFO.uniPlatform === 'web'` 时直接返回 0，这样状态栏不占高度；`getTitleBarPadding` 在非微信端返回 8px 的默认内边距；`getTitleBarHeight` 在没有胶囊按钮时，用最小高度（iOS 48px / Android 44px）兜底。最终 H5 上的导航栏会比微信小程序矮一些，只保留标题和搜索框的核心区域，视觉上更贴近普通移动端网页的头部。这样处理的结果是同一套组件在不同端呈现合理的差异化布局，而不是强行保持一致导致某端体验受损。

### 主题二：图片预加载与性能优化（高频）

#### 主问 2：壁纸预览页用 swiper 展示高清大图，如果一次性加载所有图片，内存和体验都会很差，你们是怎么优化的？

> 口播：壁纸预览是项目的核心场景，用户会快速左右滑动浏览高清壁纸。情境是一张壁纸原图可能几百 KB 到几 MB，如果列表有几十张，一次性全部加载会导致内存暴涨，低端机上直接闪退，而且初次进入预览页的白屏时间也会很长。我的任务是在保证滑动流畅的前提下，尽可能减少同时加载的图片数量。行动上，我在 `pages/preview/preview.vue` 里实现了一个 `readImgsFun` 预加载函数：维护一个 `readImgs` 响应式数组，每次 `swiperChange` 触发时，只把「当前索引、前一张、后一张」三个索引加入数组，并用 `Set` 去重。模板里每个 `swiper-item` 中的 `image` 用 `v-if="readImgs.includes(index)"` 控制，只有在这个白名单里的图片才会真正创建 DOM 和发起网络请求。对于已经加载过的图片，索引会保留在 `readImgs` 里，所以往回滑时不会重复加载。结果是预览页进入时只加载当前这张和下一张，内存占用大幅降低；滑动过程中因为前后两张已经提前加载，用户几乎感知不到白屏（待测）。

##### 追问 2.1：如果用户滑动速度很快，「前后三张」会不会出现白屏？你怎么兜底？

> 口播：快速滑动确实是一个风险点。情境是用户手指快速划过 swiper，如果滑动速度超过了图片下载和渲染的速度，下一张图可能还没加载完就切过去了，屏幕上会出现短暂空白。我的兜底有几层：第一层，`swiper` 组件开启了 `circular` 循环模式，并且 `swiper-item` 本身是全屏占位，即使图片没加载完，用户看到的是上一个页面的残留或者背景色，不会布局错乱；第二层，`readImgsFun` 在每次 `swiperChange` 时立即计算新的前后索引并加入白名单，这个计算是同步的，没有异步延迟，所以滑动一触发就开始加载；第三层，图片 URL 是从 `smallPicurl` 替换为 `.jpg` 原图，虽然是大图，但 CDN 做了图片压缩和缓存，实际加载速度在可接受范围内。如果还要进一步优化，可以考虑在 `swiperChange` 的 `onTransitionEnd` 或者 `onAnimationFinish` 里再加一个缓冲逻辑，但我目前的测试中没有收到明显的白屏反馈（待测）。

##### 追问 2.2：除了预加载，你在图片加载和渲染上还有其他优化手段吗？比如图片格式、CDN、缓存策略？

> 口播：有，而且是从数据源到展示层的一整套配合。首先，后端返回的图片列表里给的是 `smallPicurl`（缩略图，`_small.webp` 格式），我在预览页 `onLoad` 时统一把缩略图 URL 替换为 `.jpg` 原图，这样列表页加载的是小图，预览页才加载大图，避免列表页就浪费带宽。其次，图片托管在 CDN 上，CDN 本身有边缘缓存和压缩，用户二次访问同一壁纸时走的是浏览器/小程序图片缓存，不会重复下载。再次，模板里的 `image` 组件使用 `mode="aspectFill"`，让图片在保持比例的同时填满容器，减少因尺寸不匹配导致的重排。最后，`readImgs` 数组一旦加载过某个索引，就不会移除，所以用户在预览页来回滑动时，已经看过的图片是即时呈现的。这些手段叠加起来，让预览页在低端机上也能保持相对流畅的体验（待测）。

### 主题三：条件编译与多端兼容（高频）

#### 主问 3：你们项目要同时跑在微信小程序、抖音小程序、H5 和 App 上，同一份代码怎么管理平台差异？能举个例子吗？

> 口播：这是 UniApp 的核心价值，也是我在项目里花了不少精力的地方。情境是不同平台的 API 支持度、组件能力、审核规范都不一样，比如抖音小程序对自定义导航栏有限制，H5 没有 `open-type="contact"`，微信小程序有独立的广告组件。我的任务是在一套代码里优雅地处理这些差异，而不是为每个端单独维护分支。行动上，我大量使用了 UniApp 的条件编译注释。举个例子：在 `pages/index/index.vue` 中，自定义导航栏 `<custom-nav-bar>` 被包裹在 `<!-- #ifndef MP-TOUTIAO -->` 里，意思是「只要不是抖音小程序，就编译这段代码」，因为抖音小程序有自己的原生导航栏，自定义的会冲突。再比如在 `pages/preview/preview.vue` 的下载逻辑里，`#ifdef H5` 分支给用户弹 Modal 提示「请长按保存壁纸」，因为 H5 没有调用相册的权限；`#ifndef H5` 分支才是小程序和 App 的 `uni.saveImageToPhotosAlbum` 逻辑。还有 `pages/user/user.vue` 的联系客服：`#ifdef MP` 用 `<button open-type="contact">`，`#ifndef MP` 用 `uni.makePhoneCall` 拨打电话。结果是同一份代码仓库可以编译出四个端的产物，条件编译在构建期就把不需要的代码删掉，不会增加包体积。

##### 追问 3.1：条件编译是在编译期处理的，如果某段代码在运行时才发现平台不支持，你怎么兜底？

> 口播：条件编译确实是编译期行为，它解决的是「这段代码在某个平台根本不存在」的问题，比如 H5 上编译出来的代码里根本没有 `uni.saveImageToPhotosAlbum` 的调用。但运行时也会遇到平台支持了某个 API，却因为用户拒绝授权或者系统版本过低而失败的情况。我的兜底策略是分层：第一层，对于需要用户授权的操作，比如保存相册，我在 `uni.saveImageToPhotosAlbum` 的 `fail` 回调里判断 `errMsg`，如果是 `fail cancel`，Toast 提示重新点击；如果是权限拒绝，弹 Modal 引导用户去设置页开启权限。第二层，对于 API 本身可能不存在的情况，我会在调用前做能力检测，比如 `if (uni.getMenuButtonBoundingClientRect)` 再调用，不存在就走默认值兜底。第三层，在 `utils/request.js` 里统一封装了网络请求的失败处理，即使某个平台网络库行为有差异，用户看到的提示是一致的。这样编译期和运行时的兜底结合起来，能覆盖大部分平台差异导致的异常。

##### 追问 3.2：如果未来要支持一个新平台（比如快手小程序），你现在的条件编译结构容易扩展吗？

> 口播：比较容易扩展，但需要注意已有条件编译的粒度。我现在的写法大部分是平台正向判断（`#ifdef MP-WEIXIN`）或者反向排除（`#ifndef MP-TOUTIAO`）。如果要支持快手小程序（假设平台标识是 `mp-kuaishou`），我需要审视所有条件编译块：对于「只在微信小程序做」的逻辑（比如广告位），如果快手也支持类似广告组件，我可以把 `#ifdef MP-WEIXIN` 改成 `#ifdef MP-WEIXIN || MP-KUAISHOU`；对于「排除抖音」的逻辑（`#ifndef MP-TOUTIAO`），快手通常不受影响，可以直接编译进去。风险在于某些 `#ifndef H5` 的块里可能隐含了「小程序通用」的假设，如果快手小程序的 API 行为与微信有差异，可能需要进一步拆分。为了降低扩展成本，我在 `utils/system.js` 里抽象了 `getMpMenuRect` 这类函数，把平台差异收敛到工具层，页面层只调用函数而不直接写条件编译，这样新增平台时只需要改工具层的兜底值，页面层基本不用动。

### 主题四：本地缓存与页面间数据传递（中高频）

#### 主问 4：小程序页面之间传参有长度限制，你们壁纸列表数据量又不小，是怎么把列表数据带到预览页的？

> 口播：这是一个很典型的跨页数据传递问题。情境是分类列表页（`classlist`）有分页加载，一页 12 条，滑几页后可能有几十条数据，每条数据包含 `_id`、`picurl`、`smallPicurl`、评分等字段。如果通过 URL `query` 传 `id` 数组，长度很容易超过小程序路由限制，而且预览页需要知道整条列表的顺序，才能支持左右滑动切换。我的任务是在不二次请求的前提下，把完整列表带给预览页。行动上，我选择了 `uni.setStorageSync` 作为数据通道：在 `pages/classlist/classlist.vue` 里，每次加载或追加数据后，执行 `uni.setStorageSync('storgClassList', classList.value)`；用户点击某张壁纸进入预览页时，只传当前壁纸的 `id` 和 `previewList` 标识；预览页 `onLoad` 中通过 `uni.getStorageSync(previewList)` 读取完整列表，再根据 `id` 找到当前索引。页面卸载时，`onUnload` 里调用 `uni.removeStorageSync(previewList)` 清理缓存，避免残留。结果是预览页打开时无需等待接口，滑动切换是本地数据驱动，体验很流畅；同时缓存及时清理，不会无限膨胀（待测）。

##### 追问 4.1：如果用户从分享链接直接进入预览页，没有经历过列表页，Storage 里没有数据，你怎么处理？

> 口播：分享场景确实会绕过列表页。情境是用户 A 把某张壁纸分享给用户 B，用户 B 点击分享卡片直接进入 `pages/preview/preview?id=xxx&type=share`，此时 `Storage` 里没有 `storgClassList`。我的处理是在预览页 `onLoad` 里加一个分支：如果 `e.type == 'share'`，就调用 `apiDetailWall({ id: currentId.value })` 接口，把当前壁纸的详情请求回来，并构造一个只包含这张壁纸的 `classList`。这样用户 B 至少能看到这张壁纸的全屏预览，虽然无法左右滑动浏览更多（因为不知道上下文列表），但核心功能不受影响。如果业务上希望分享页也能滑动，可以进一步改造：让后端在详情接口里返回「相关推荐」列表，或者前端根据 `classid` 再请求同分类的壁纸列表作为上下文。目前项目中采用的是单张兜底方案，保证分享链路不会白屏。

##### 追问 4.2：Storage 有容量限制（微信小程序一般是 10MB），你们怎么避免缓存撑满？

> 口播：微信小程序的 Storage 单上限确实是 10MB，而且壁纸列表数据如果包含大量 URL 字符串，累积起来不可忽视。我的规避策略有几层：第一层，控制缓存内容，我只存壁纸列表的元数据（`_id`、`picurl`、`smallPicurl` 等），不存图片二进制，实际占用很小；第二层，及时清理，预览页 `onUnload` 里一定调用 `uni.removeStorageSync(previewList)`，确保离开预览页就释放；搜索页 `onUnload` 里也清理了 `storgClassList`。第三层，避免多份重复缓存，列表页和搜索页都用同一个 key `storgClassList`，新数据直接覆盖旧数据，不会累积多份。第四层，如果未来数据量真的很大，可以考虑只缓存当前页和前后页的 id 数组，进预览页后再根据 id 分批请求详情，但目前的元数据体积远未到需要这一步的程度（待测）。

### 主题五：请求封装与错误处理（中高频）

#### 主问 5：你们项目的接口请求是怎么封装的？如果某个接口报错，用户端的表现是什么？

> 口播：我在项目里封装了一个通用的 `request` 模块，路径是 `utils/request.js`。情境是如果每个页面都直接写 `uni.request`，不仅重复代码多，而且 header 里的鉴权字段、错误提示逻辑很难统一，一旦后端改了错误码规范，要改很多地方。我的任务是把请求收敛到一个入口，统一处理成功、业务错误和系统错误。行动上，我封装了一个返回 Promise 的 `request` 函数：入参解构出 `url`、`data`、`method`、`header`，自动拼接 `BASE_URL`，并注入 `access-key` header。在 `uni.request` 的 `success` 回调里，我按 `res.data.errCode` 分流：`errCode === 0` 时 `resolve(res.data)`，表示业务成功；`errCode === 400` 时 `uni.showModal` 弹窗提示错误内容，并 `reject(res.data)`，因为 400 一般是参数错误或重要业务阻断，需要用户明确感知；其他错误码用 `uni.showToast` 轻量提示，并 `reject`。`fail` 回调里直接 `reject(err)`，交给调用方捕获。结果是所有页面的请求都走这一个入口，错误提示风格统一；调用方可以用 `try/catch` 做二次兜底，比如预览页评分失败时隐藏 loading 并捕获异常。

##### 追问 5.1：如果后端返回了 400 错误，你弹了 Modal，但用户正在输入表单，Modal 打断体验怎么办？

> 口播：这是一个体验取舍的问题。情境是 400 错误通常代表「业务规则不通过」，比如参数缺失、重复评分、超出限制等，这类错误需要用户明确知道「为什么失败了」，所以用 Modal 比 Toast 更合适，因为 Toast 停留时间短，用户可能没看清。但如果是在表单输入过程中触发 400，Modal 确实会打断当前输入流。我的处理是：在调用层（比如 `submitScore`）里，我在发起请求前做了前置校验，比如 `userScore` 为 0 时直接禁用提交按钮，避免无意义请求；请求过程中用 `uni.showLoading` 罩住界面，防止用户重复点击；如果最终还是触发了 400，Modal 弹出后用户点击确定，界面会回到可操作状态，输入的内容不会丢失，因为 Vue 的响应式数据不会因为弹窗而重置。如果未来有更高体验要求，可以把 Modal 换成页面内嵌的错误提示区域，但目前 Modal 的明确性对壁纸这类轻量应用是够用的。

##### 追问 5.2：你们有做请求拦截器吗？比如统一加 loading、统一记日志、token 过期自动刷新？

> 口播：目前项目里的 `utils/request.js` 是一个轻量封装，还没有做到 axios 那种完整的拦截器链。但已经具备了拦截器的雏形：统一加 header（`access-key`）和统一错误处理可以看作「响应拦截」的一层。对于 loading，我没有在 `request.js` 里统一加，因为不同页面的交互不一样：有的页面需要全屏 loading（比如评分提交），有的只需要骨架屏（比如专题列表），所以 loading 的控制权放在调用方。对于 token 过期自动刷新，这个项目使用的是固定 `access-key` 的鉴权方式，没有用户登录态 token，所以暂时没有刷新机制。如果未来接入用户登录，我计划在 `request.js` 里增加请求拦截器：在发送前检查 token 有效期，如果临近过期先调刷新接口，再重放原请求；响应拦截器里捕获 401 状态码，清除本地登录态并引导重新授权。这些扩展在现在的封装结构上是很容易加进去的，因为所有请求都收敛在一个函数里。

### 主题六：分享链路、广告位与商业化（中频）

#### 主问 6：小程序的分享功能你们是怎么做的？预览页分享出去后，其他用户打开能看到什么？

> 口播：分享是小程序裂变的核心能力，我在项目里多个页面都配置了分享。情境是用户看到喜欢的壁纸，想分享给好友或者发朋友圈，分享出去后接收方需要能直接看到这张壁纸，而不是跳到一个无关的页面。行动上，我在首页、分类列表、预览页、排行榜、近期上新等核心页面都实现了 `onShareAppMessage`（分享给好友）和 `onShareTimeline`（分享朋友圈）。以预览页为例：`onShareAppMessage` 返回的 `path` 是 `/pages/preview/preview?id=当前壁纸ID&type=share`，`title` 是「咸虾米壁纸」；`onShareTimeline` 返回的 `query` 也带同样的参数。接收方打开后，预览页 `onLoad` 识别到 `type=share`，会调用 `apiDetailWall` 接口把这张壁纸的详情拉回来，构造单张壁纸的列表，保证用户能正常预览。首页和分类列表的分享则带分类 ID 和名称，接收方打开后直接进入对应的列表页，保持上下文一致。结果是分享链路完整闭环，从任意页面分享出去都能恢复到合理的页面状态。

##### 追问 6.1：分享朋友圈和分享好友在参数传递上有什么区别？你为什么要区分处理？

> 口播：两者的参数传递机制确实不一样，这是小程序平台的限制决定的。`onShareAppMessage`（好友）是通过 `path` 字段带参，格式是完整的页面路径加 query，比如 `/pages/preview/preview?id=123&type=share`；而 `onShareTimeline`（朋友圈）只能通过 `query` 字段带参，格式是 `id=123&type=share`，不能带路径。如果我在朋友圈分享里错误地用了 `path`，参数会丢失，接收方打开后 `onLoad` 拿不到 `id`，就会白屏或者进入默认状态。所以我区分处理：好友分享用 `path` 拼完整路径，朋友圈分享用 `query` 拼参数字符串，但两者携带的业务参数是一致的。在预览页 `onLoad` 里，我通过 `e.id` 和 `e.type` 统一读取，不管从哪个渠道进来都能正确解析。

##### 追问 6.2：近期上新页面里插入了广告位，广告是怎么动态插入到内容流里的？怎么控制只在微信小程序展示？

> 口播：广告位的动态插入是近期上新模块的一个商业化需求。情境是近期上新是一个瀑布流列表，业务希望在第 10 个位置插入一条视频广告，而且广告组件 `<ad>` 只在微信小程序平台有效，其他平台编译会报错。行动上，我在 `pages_quick/news.vue` 的 `getList` 函数里，请求到数据后做了一道处理：`#ifdef MP-WEIXIN` 条件下，如果 `data.length > 9`，就在索引 9 的位置 `splice(9, 0, { ad: true })`，插入一个标记对象。模板渲染时，`v-for` 遍历 `listData`，如果 `item.ad` 为 true，就渲染 `<ad unit-id="...">` 广告组件；否则渲染正常的壁纸 `picture-item`。为了让广告通栏展示，我给广告项加了 `aditem` class，用 `grid-column: 1/4` 让广告跨越三列网格，占据整行。对于非微信端，条件编译在构建期就把这段广告代码删掉了，不会报错也不会占位。结果是微信用户能看到列表中自然插入的广告，其他平台用户完全无感知，列表布局也不会错乱（待测）。

### 主题七：用户交互与权限处理（中频）

#### 主问 7：用户点击下载壁纸时，如果拒绝授权保存到相册，你们是怎么引导的？

> 口播：下载保存是壁纸应用的核心功能，但小程序的相册授权是用户敏感权限，很容易被拒绝。情境是用户点击下载按钮后，我调用 `uni.saveImageToPhotosAlbum`，系统会弹出授权框，如果用户点了拒绝，后续再调用会直接失败，而且不会再弹授权框。我的任务是在授权失败后，给用户一个清晰的挽回路径。行动上，我在 `saveImageToPhotosAlbum` 的 `fail` 回调里做了分支判断：如果 `errMsg` 是 `saveImageToPhotosAlbum:fail cancel`，说明用户主动取消了，我 Toast 提示「保存失败，请重新点击下载」；如果是权限拒绝（其他 errMsg），我弹一个 `uni.showModal`，标题是「授权提示」，内容是「需要授权保存相册」，用户点击确定后，调用 `uni.openSetting` 打开小程序设置页，让用户手动开启「保存到相册」权限。在 `openSetting` 的 `success` 回调里，我再检查 `authSetting['scope.writePhotosAlbum']` 是否开启，开启就 Toast 提示成功，没开启就提示失败。结果是即使用户第一次拒绝了，也有一条明确的路径可以重新授权，而不是功能直接不可用。

##### 追问 7.1：H5 端没有 `saveImageToPhotosAlbum` 这个 API，你们的下载逻辑是怎么兼容的？

> 口播：H5 的下载逻辑和小程序完全不同。情境是 H5 运行在浏览器里，浏览器没有「保存到系统相册」的概念，而且跨域图片直接触发下载也有很多限制。我的处理方式是用条件编译彻底分离两条链路：在 `pages/preview/preview.vue` 的 `clickDownload` 函数里，`#ifdef H5` 分支直接弹 `uni.showModal`，提示「请长按保存壁纸」，因为 H5 上用户长按图片，浏览器通常会弹出「保存图片」的菜单，这是最直接的用户习惯；`#ifndef H5` 分支才是小程序和 App 的 `uni.saveImageToPhotosAlbum` 逻辑。这样 H5 用户不会走到一个不存在的 API 调用里，也不会看到莫名其妙的报错。如果未来 H5 需要更主动的下载能力，可以考虑用 `<a download>` 标签或者 `canvas` 绘制后转 blob 下载，但目前长按保存的方案简单可靠，符合用户预期。

##### 追问 7.2：评分功能里，用户已经评过分的壁纸，再次点击评分按钮会发生什么？

> 口播：评分功能有防重复评分的机制。情境是用户给某张壁纸评过分后，再次进入预览页点击评分按钮，如果还能重新评，会导致数据混乱和用户体验不佳。行动上，我在 `clickScore` 函数里做了判断：如果 `currentInfo.value.userScore` 存在（说明当前壁纸对象里已经记录了用户的评分），就把 `isScore` 设为 true，并把 `userScore` 回显为已评分值；弹窗标题也会变成「评分过了~」。同时，评分弹窗里的 `uni-rate` 组件设置了 `:disabled="isScore"`，已评分状态下星星不可点击；提交按钮也加了 `:disabled="!userScore || isScore"`，确保已评分时无法再次提交。如果用户从未评过，`isScore` 为 false，星星可交互，提交按钮在用户选择分数后启用。这样避免了重复评分，也让用户能清晰看到自己之前的评分结果。

### 主题八：工程化与代码组织（中频）

#### 主问 8：你们项目的 API 接口是怎么组织的？如果后端新增了一个接口，你需要改哪些地方？

> 口播：我在项目里把 API 集中管理在 `api/apis.js`，这是工程化里很基础但很重要的一点。情境是如果接口分散在各个页面里，后端改了路径或参数，要全局搜索替换，很容易漏改。行动上，`api/apis.js` 统一导入 `utils/request.js` 里的 `request` 函数，然后为每个业务接口导出一个函数，比如 `apiGetBanner`、`apiSearchData`、`apiWriteDownload` 等，这些函数只做一件事：调用 `request` 并传入对应的 `url` 和 `data`。页面层只导入需要的 API 函数，不直接写 `uni.request`。如果后端新增一个接口，我只需要在 `api/apis.js` 里新增一个导出函数，然后在对应的页面里导入调用即可，不需要改 `request.js` 的封装逻辑。如果后端改了某个接口的路径，我也只需要改 `api/apis.js` 里对应函数的那一行 `url`，所有调用这个函数的页面自动生效。结果是接口管理非常集中，维护和迭代成本很低。

##### 追问 8.1：`utils/common.js` 和 `utils/tools.js` 都放了工具函数，你是怎么决定一个函数该放哪个文件的？

> 口播：这是我在项目初期就定下来的分层约定。`utils/common.js` 放的是「与业务强相关、与 UniApp API 强耦合」的函数，比如 `routerTo`（封装了 `uni.navigateTo` / `uni.reLaunch` 的降级逻辑）、`goBack`（返回失败时自动回首页）、`showToast`（统一 Toast 配置）、`gotoHome`（页面异常时引导回首页）。这些函数的特点是直接调用 `uni.xxx` API，而且和业务场景紧密绑定，放到 `common.js` 里供页面层直接使用。`utils/tools.js` 放的是「纯数据转换、与业务无关」的通用工具，比如 `formatNumber`（数字转 w/k 单位）、`getDateRange`（返回近 7 天日期范围）、`formatUpdateTime`（时间戳转相对时间）。这些函数不依赖 UniApp，拿到其他项目也能用。这样分层的目的是：如果未来项目要抽成组件库或者迁移到非 UniApp 框架，`tools.js` 可以直接搬走，`common.js` 则需要根据新框架的 API 做适配。

##### 追问 8.2：你们有做组件拆分吗？能举一个你觉得拆分得最合理的组件例子吗？

> 口播：有，而且我觉得拆分得最合理的是 `components/custom-nav-bar/custom-nav-bar.vue`。情境是项目里多个页面需要自定义导航栏，如果每个页面都自己写一套状态栏+标题栏+填充块的逻辑，代码会大量重复，而且一旦要改导航栏样式（比如加搜索框、改背景色），要改很多地方。行动上，我把导航栏抽象成一个组件，接收 `title` prop，内部自动计算状态栏高度、标题栏高度，并渲染搜索框入口。页面层只需要引入组件并传一个标题，下方自动有填充块占位。更关键的是，这个组件内部已经处理了抖音小程序的隐藏逻辑（`#ifndef MP-TOUTIAO`），调用方完全不用关心平台差异。另一个拆得比较好的是 `picture-item`，它在列表页、搜索页、分类页、排行榜等多个场景复用，统一处理图片点击跳转预览的逻辑，并负责从 `classList` 中读取数据传入 Storage。这些组件的共同点是把「跨页面复用」和「平台差异封装」收敛到内部，页面层只关注业务数据。
